# Data Provenance and Research Integrity

This document describes the chain of custody for all data used in the study,
why the analytics database must be reset before real data collection, and how
simulation data from the prototype phase is distinguished from research data.

---

## 1. Two Distinct Phases — Two Separate Datasets

| Phase | Period | Data | Status |
|-------|--------|------|--------|
| **NCCIT 2026 prototype** | Nov 2025 – Apr 2026 | n ≈ 32, single institution, 1 round | Preliminary — must be cited separately as "NCCIT2026 prototype (n=32)" |
| **Extended study (IEEE TLT)** | Jun 2026 – Aug 2026 | n = 60–70 target, 3 rounds, IRB-approved | Not yet collected — pending IRB approval |

These two datasets are **not interchangeable**. Results from the prototype
cannot be reported as results from the extended study, and vice versa.

The analytics database currently contains a **mix of prototype data and
simulation data** (see Section 3). It must be reset before the extended
study begins.

---

## 2. Chain of Custody (Extended Study)

Every record in the final research dataset must trace back through this chain:

```
IRB approval (KMUTNB)
    ↓
Participant enrollment (CTFd account creation)
    ↓
experiment.assign_participant(user_id)   ← records in experiment_assignment table
    ↓
Platform usage generates events          ← events table (append-only, no UPDATE/DELETE)
    ↓
data_collection_log records milestones   ← round start/end, participant counts
    ↓
tools/research/verify_data_provenance.py ← gate: refuses if any user lacks assignment
    ↓
tools/research/export_anonymized_dataset.py ← scrubs PII, assigns P001–P0XX codes
    ↓
tools/research/generate_manifest.py     ← SHA-256 hashes + DB stats → manifest.json
    ↓
tools/analysis/statistical_tests.py     ← analysis (also gated by provenance check)
```

Every step is verifiable: the `experiment_assignment` table records who was
enrolled when, the `events` table is append-only (trigger prevents alteration),
and `manifest.json` cryptographically links the final CSV files to the DB state
at export time.

## 3. Simulation Data — History and Permanent Exclusion

During development of the NCCIT prototype, a simulation script
(`simulate_round_comparison_cohort.py`) injected synthetic events into the
analytics database to test the round comparison dashboard. This script created:

| Simulated user pattern | Seed | Purpose |
|------------------------|------|---------|
| `bank.s`, `bank.s01–03` | 20260407 | "impulsive" behavioral trajectory |
| `mint.s01–03` | 20260407 | "cautious" behavioral trajectory |
| `oak.s01–03` | 20260407 | "protocol-first" behavioral trajectory |

These users have **never been real research participants**. They are
permanently excluded from research analysis by:

1. `verify_data_provenance.py` pattern-matching against `bank.s*`, `mint.s*`, `oak.s*`
2. `reset_for_data_collection.sh` wiping the DB before the extended study begins
3. The simulation script being moved to `tests/fixtures/legacy_simulation/` with
   a warning that prohibits running it against the research database

---

## 4. Pre-Collection Checklist

Before enrolling any real participant:

- [ ] IRB approval from KMUTNB Human Research Ethics Committee received
- [ ] `tools/setup/reset_for_data_collection.sh` run (typed "RESET" to confirm)
- [ ] `verify_data_provenance.py` reports 0 simulation, 0 unknown, 0 total users
- [ ] `data_collection_log` shows `db_initialized` entry with timestamp
- [ ] At least one test enrollment created and verified via `experiment.assign_participant()`
- [ ] Backup policy confirmed (`scripts/backup_db.sh` scheduled via cron)

---

## 5. What "Anonymized" Means in This Study

The public dataset (generated by `export_anonymized_dataset.py`) replaces all
direct identifiers with stable participant codes (P001, P002, …):

- No name, email, username, IP address, or institutional identifier
- Timestamps converted to `ts_offset_seconds` (relative, not wall-clock)
- Open-text survey responses passed through PII scrubber (regex: email, Thai phone, national ID, name prefixes)
- The private re-identification map (`data/_code_map_private.csv`) is never committed to version control

---

## 6. NCCIT Prototype Data — If Used

If any results from the NCCIT 2026 prototype phase are cited in the
manuscript, they must be:

- Clearly labeled as **"NCCIT2026 prototype (n=32)"** in all tables and figures
- Described in the methodology as a **separate preliminary study**, not the main study
- Not pooled with extended study data in any inferential analysis
- Accompanied by a statement that the prototype predates IRB approval for the extended study
